# Launch Plan

This file turns the current repository state into an English-first launch sequence.

## Goal

Ship `VIBE-Claude-Plugin` as a clear open source project for GitHub and English-speaking developer communities.

The launch goal is not raw reach alone. The goal is qualified attention:

- people understand what the project does in under 10 seconds
- interested users can install it quickly
- early feedback turns into repo improvements

## Positioning

Use this positioning everywhere unless there is a strong reason to narrow it:

`VIBE-Claude-Plugin is a Claude Code plugin that turns prompt-based coding into a document-driven, multi-stage engineering workflow.`

## Primary Channels

Focus on these first:

1. GitHub repository
2. X
3. Reddit
4. Product Hunt
5. Dev.to

Chinese communities can follow later as a secondary wave.

## Launch Order

### Step 1: Tighten the Repository

Before posting anywhere:

- keep `README.md` English-first
- keep `/vibe` as the public command in every document
- keep `vibe-autopilot` described as the internal workflow
- add one demo GIF or short video near the top of `README.md`
- prepare one real case study

### Step 2: Publish the Core Assets

Have these ready before public launch:

- repository URL
- short demo video or GIF
- one screenshot of requirement alignment
- one screenshot of generated artifacts such as `tasks.md`
- one screenshot of implementation or review flow

### Step 3: Launch on X

Use X as the first public signal.

Post format:

- one short opener
- one screenshot or GIF
- one reply with the repo link

Target message:

- what problem Claude Code has on larger tasks
- what this plugin adds
- why `/vibe` is different from a plain prompt

### Step 4: Launch on Reddit

Good targets:

- `r/ClaudeAI`
- `r/LocalLLaMA` if framed around workflow rather than hype
- `r/SideProject`
- relevant coding agent subreddits if they allow launch posts

Post style:

- explain the problem first
- describe the workflow concretely
- include screenshots
- expect skeptical comments and answer with specifics

### Step 5: Launch on Product Hunt

Use Product Hunt only after the repository and demo assets look credible.

Need before launch:

- polished tagline
- short description
- logo or clean visual
- 3 to 5 screenshots
- maker comment

### Step 6: Publish a Dev.to Article

Turn the launch into a more durable article:

- what breaks in long coding sessions
- why a document-driven workflow helps
- how `VIBE-Claude-Plugin` structures delivery
- where it still has limits

## First-Week Schedule

### Day 1

- finalize README
- prepare demo GIF or video
- prepare screenshots

### Day 2

- post on X
- post on one Reddit community

### Day 3

- collect feedback
- fix onboarding friction
- update README if users get stuck

### Day 4

- publish Dev.to article
- post a second X update with one real example

### Day 5 to Day 7

- prepare Product Hunt launch if early feedback is strong
- convert repeated questions into FAQ or setup fixes

## Metrics To Track

Track simple signals first:

- GitHub stars
- repository traffic
- README click-through to install steps
- issues opened
- successful installs reported by users

Do not over-optimize for impressions without activation.

## Messaging Rules

- say exactly what the project does
- avoid promising full autonomy for every project
- describe `/vibe` as the public command
- describe `vibe-autopilot` as the internal workflow
- show the process, not just the claim

## What To Improve After Launch

Once the first feedback round lands, improve in this order:

1. installation clarity
2. demo quality
3. real case studies
4. stronger proof of testing and review flow
5. onboarding for first-time Claude Code users
