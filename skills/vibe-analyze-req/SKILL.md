# 项目需求分析工作流

## 1. 文档目的
规范项目需求分析的流程，确保AI能按用户要求生成完整的PDR需求文档，包含技术栈等关键信息。

## 2. 工作流结构

一个简洁高效的项目需求分析工作流应包含：

- **基本信息**：项目名称、分析目标
- **前置步骤**：准备工作、沟通准备
- **执行步骤**：详细的需求分析流程和顺序
- **执行建议**：专业建议和注意事项
- **成功标准**：需求分析完成的判定条件
- **失败处理**：异常情况的应对措施
- **输出成果**：明确的交付物和保存位置

## 3. 执行要求

- 严格按照步骤执行
- 与用户保持深度沟通
- 记录关键信息和结果
- 遇到异常时按失败处理机制执行
- 确保输出成果符合用户预期

## 4. 文档管理

- 执行后根据实际情况更新文档
- 进行版本管理，确保使用最新版本


## 工作流程

### 前置步骤：准备工作
1. **沟通环境准备**：
   - 确保与用户的沟通渠道畅通
   - 准备好需求收集的工具和文档模板
   - 设定合理的沟通时间和频次

2. **分析准备**：
   - 了解项目的基本背景和行业领域
   - 准备行业相关的技术栈和最佳实践知识
   - 制定详细的需求收集计划

### 第一步：与用户沟通项目基本信息
1. **项目概览确认**：
   - 与用户沟通："您希望开发的项目是什么类型？请简要描述项目的核心功能和目标。"
   - 了解项目的业务领域和应用场景
   - 记录项目的基本信息和初步定位

2. **项目目标确认**：
   - 与用户沟通："您希望通过这个项目解决什么问题？达到什么业务目标？"
   - 明确项目的核心价值和成功指标
   - 了解项目的预期成果和影响范围

3. **项目约束确认**：
   - 与用户沟通："项目有哪些时间、预算、技术或资源方面的约束？"
   - 了解项目的限制条件和边界
   - 记录可能影响项目范围的因素

### 第二步：与用户沟通功能需求
1. **核心功能确认**：
   - 与用户沟通："项目的核心功能模块有哪些？每个模块的主要功能是什么？"
   - 详细了解每个功能模块的具体需求
   - 记录功能的优先级和重要程度

2. **用户流程确认**：
   - 与用户沟通："用户将如何使用这个系统？请描述主要的用户操作流程。"
   - 了解系统的使用场景和用户交互流程
   - 绘制初步的用户流程图

3. **功能细节挖掘**：
   - 与用户沟通："每个功能模块有哪些具体的操作和交互细节？"
   - 深入了解功能的具体实现要求
   - 记录功能的边界条件和特殊情况

### 第三步：与用户沟通技术需求
1. **技术栈偏好确认**：
   - 与用户沟通："您对项目的技术栈有什么偏好或要求？"
   - 了解用户对前端、后端、数据库等技术的偏好
   - 记录技术选型的限制条件

2. **性能要求确认**：
   - 与用户沟通："系统需要支持多少并发用户？响应时间有什么要求？"
   - 了解系统的性能指标和预期负载
   - 记录性能相关的技术要求

3. **安全需求确认**：
   - 与用户沟通："系统需要考虑哪些安全方面的需求？"
   - 了解数据安全、用户认证、权限管理等安全要求
   - 记录安全相关的技术标准和合规要求

4. **集成需求确认**：
   - 与用户沟通："系统需要与哪些现有系统或第三方服务集成？"
   - 了解系统的外部接口和集成要求
   - 记录集成相关的技术细节

### 第四步：与用户沟通非功能需求
1. **用户体验要求**：
   - 与用户沟通："您对系统的用户体验有什么要求？"
   - 了解界面设计风格、交互方式、响应速度等要求
   - 记录用户体验相关的设计规范

2. **可扩展性要求**：
   - 与用户沟通："系统未来可能需要扩展哪些功能？"
   - 了解系统的可扩展性需求和未来规划
   - 记录系统架构的灵活性要求

3. **可维护性要求**：
   - 与用户沟通："系统的维护和更新有什么要求？"
   - 了解系统的部署、监控、日志等运维需求
   - 记录系统的可维护性标准

### 第五步：需求分析与文档生成
1. **需求整理与分析**：
   - 整理所有收集到的需求信息
   - 分析需求的可行性和合理性
   - 识别需求之间的依赖关系和潜在冲突

2. **技术方案设计**：
   - 基于需求分析，设计初步的技术方案
   - 选择适合的技术栈和架构模式
   - 评估技术方案的可行性和风险

3. **PDR需求文档生成**：
   - 使用PDR模板，生成完整的需求文档
   - 确保文档包含：项目概览、功能需求、技术需求、非功能需求、技术栈选择、实施计划等关键信息
   - 对文档内容进行审核和校对

4. **文档确认**：
   - 与用户沟通："请审核生成的PDR需求文档，确认是否符合您的预期？"
   - 根据用户反馈，对文档进行修改和完善
   - 确保文档内容准确反映用户的需求和期望

### 第六步：需求管理与后续跟进
1. **需求变更管理**：
   - 建立需求变更的流程和机制
   - 明确需求变更的评估和审批流程
   - 记录需求变更的历史和影响

2. **后续跟进计划**：
   - 与用户沟通后续的开发和实施计划
   - 确定需求文档的使用方式和维护责任
   - 安排定期的需求回顾和更新

## 执行建议
1. **深度沟通**：与用户进行深入对话，确保全面了解项目需求
2. **专业引导**：基于技术和行业经验，引导用户思考需求的完整性和可行性
3. **系统化分析**：采用结构化的方法收集和分析需求，确保不遗漏关键信息
4. **技术与业务结合**：平衡技术可行性和业务需求，提供合理的技术建议
5. **文档质量**：确保PDR文档内容完整、准确、清晰，便于后续开发和沟通
6. **持续沟通**：在整个需求分析过程中保持与用户的持续沟通，及时解决疑问和分歧
7. **风险识别**：在需求分析过程中识别潜在的风险和挑战，提前制定应对措施
8. **可追溯性**：确保每个需求都有明确的来源和依据，便于后续的需求验证
9. **灵活性**：在保持需求完整性的同时，为后续的调整和优化预留空间
10. **用户参与**：鼓励用户积极参与需求分析过程，确保最终文档符合用户的真实需求

## 输出成果
1. 完整的PDR需求文档（包含项目概览、功能需求、技术需求、非功能需求、技术栈选择、实施计划等关键信息） - 保存至`./项目文档/需求文档/`目录
2. 技术方案设计文档（包含技术栈选择、架构设计、关键技术点等） - 保存至`./项目文档/技术文档/`目录

记住，完整准确的需求分析是项目成功的基础。通过系统化的流程和深度的沟通，确保生成的PDR需求文档能够准确反映用户的需求，为后续的开发和实施提供清晰的指导。
